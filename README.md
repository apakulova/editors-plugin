# Чистовик

«Чистовик» — Figma-плагин для быстрой чистки русской типографики в текстовых слоях. Он приводит кавычки, тире, пробелы, числа, телефоны, сокращения и базовые символы к единым правилам, не создавая новые текстовые слои и не удаляя старые.

Плагин рассчитан на редакторскую проверку макетов: можно запустить его на одном текстовом слое, на группе/фрейме/компоненте или сразу на всей текущей странице.

## Команды

- `Запустить типограф` — сразу запускает обработку в основном режиме без интерфейса.
- `⭐ Продвинутые фичи` — открывает окно настроек. В интерфейсе есть вкладки `Типограф`, `Правила` и `О плагине`. Рабочий запуск типографа находится на вкладке `Типограф`.

## Что важно

- Плагин изменяет существующие текстовые ноды, а не пересоздает их.
- Текст обновляется через `textNode.characters = newText`.
- Перед изменением плагин загружает используемые шрифты.
- Смешанные стили сохраняются и восстанавливаются по диапазонам: шрифт, размер, заливки, регистр, подчеркивание/зачеркивание, трекинг, высота строки, списки, отступы и абзацные интервалы.
- Переносы строк сохраняются: абзацы не склеиваются.
- Заблокированные слои не обрабатываются по умолчанию. Если замок стоит на текстовом слое, фрейме, группе, секции или другом родителе, текст внутри пропускается. В UI есть чекбокс `Обработать даже слои с замочком`.
- Скрытые слои не обрабатываются по умолчанию. Если скрыт текстовый слой или его родитель, текст внутри пропускается. В UI есть чекбокс `Обработать даже скрытые слои`.
- В `manifest.json` включен `documentAccess: "dynamic-page"`, чтобы Figma не загружала все страницы файла при открытии настроек или быстром запуске.
- В `manifest.json` разрешен сетевой доступ только к `https://eu.i.posthog.com`, чтобы отправлять анонимные технические события аналитики.
- Библиотека `typograf` не используется, чтобы встроенные правила не конфликтовали с кастомной логикой.

Если в тексте много ручного форматирования внутри одного слоя, лучше сначала проверить плагин на копии фрейма. В Figma также всегда можно откатить результат через Undo.

## Анонимная аналитика

Плагин отправляет в PostHog Cloud EU анонимные технические события использования. Аналитика нужна, чтобы понимать количество уникальных пользователей, ежедневные запуски, популярность режимов, использование настроек обработки замочков и скрытых слоев, тип области запуска, клики по ссылке на канал и ошибки плагина.

Аналитика встроена по умолчанию и не отключается в интерфейсе. Она не должна влиять на скорость работы, уведомления и результат типографики. Если PostHog недоступен или запрос аналитики завершается ошибкой, плагин должен работать как раньше.

Отправляются только технические события и агрегированные свойства:

- открытие окна настроек;
- старт, успешное завершение и ошибка запуска типографа;
- источник запуска: быстрый запуск или запуск из настроек;
- режим запуска: по умолчанию, `Для красоты` или `Для разработки`;
- флаги `processLockedNodes`, `processHiddenNodes` и `recolorExistingAsterisks`;
- тип области запуска: один текстовый слой, контейнер, вся страница или мультивыбор;
- счетчики обработанных, измененных, пропущенных заблокированных и пропущенных скрытых текстовых слоев;
- счетчик найденных текстовых слоев и количество текстовых нод с ошибкой обработки;
- длительность обработки;
- клик по ссылке на канал;
- технический fingerprint ошибки без текста макета.

Не отправляются:

- текст макета;
- названия файлов;
- названия страниц;
- названия слоев;
- email;
- имя пользователя;
- Figma user id;
- полный stack trace.

Для событий дополнительно отключается GeoIP-обогащение PostHog, чтобы сервис не добавлял к событиям город, страну, координаты и другие свойства, рассчитанные по IP.

Для уникальности используется анонимный случайный идентификатор из `figma.clientStorage`. Он не привязан к личности пользователя и может сброситься при очистке кэша, смене устройства или смене `plugin id`. Если в будущем появится платный тариф и добровольная привязка email, ее нужно реализовывать отдельным этапом и не отправлять email в аналитику открытым текстом.

События передаются в PostHog Capture API без пользовательского поля `uuid`: PostHog принимает только отсутствующий `uuid` или настоящий UUID-формат, а внутренние id вида `evt_...` использовать в payload нельзя. Для локальной надежности у плагина есть очередь событий в `figma.clientStorage`; при чтении старые события из очереди нормализуются перед отправкой, чтобы legacy-payload не блокировал новые события. Аналитика не должна задерживать UI-сценарии: запуск из настроек не ждет отправку событий, а быстрый запуск перед закрытием дает аналитике только короткий grace period.

Рабочий план внедрения и будущего развития аналитики хранится в `docs/analytics-plan.md`.

## Ежедневный Telegram-отчет

Ежедневная отправка работает через Vercel Cron и endpoint `api/daily-analytics.js`. Vercel один раз в день вызывает `/api/daily-analytics`, endpoint проверяет `CRON_SECRET`, собирает сводку PostHog за предыдущий календарный день по Москве и отправляет ее в Telegram.

В `vercel.json` задан cron `0 6 * * *`, то есть `09:00` по Москве. На бесплатном Hobby-тарифе Vercel может выполнить cron не ровно в указанную минуту, а в пределах часа.

Сообщение отправляется на русском языке. Если за день нет событий, отчет все равно отправляется с нулями.

Для ежедневного отчета во Vercel должны быть добавлены Environment Variables:

- `POSTHOG_PERSONAL_API_KEY` — PostHog Personal API key для чтения аналитики;
- `POSTHOG_HOST` — `https://eu.posthog.com`;
- `POSTHOG_PROJECT_ID` — `184090`;
- `POSTHOG_DASHBOARD_URL` — `https://eu.posthog.com/project/184090/dashboard/695809`;
- `TELEGRAM_BOT_TOKEN` — токен Telegram-бота;
- `TELEGRAM_CHAT_ID` — id чата, куда бот отправляет отчет;
- `CRON_SECRET` — длинная случайная строка для защиты cron endpoint.

GitHub Actions workflow `.github/workflows/daily-analytics.yml` оставлен только для ручного резервного запуска через `workflow_dispatch`; расписание из него отключено, чтобы ежедневные сообщения не дублировались.

Скрипт отчета лежит в `scripts/send-daily-analytics.js`. Он использует общую логику из `scripts/lib/analytics-report.js`: исключает тестовые события с `is_test_event: true`, считает основные метрики по событиям `plugin_run_started`, `plugin_run_completed`, `plugin_run_failed`, `settings_opened` и `channel_link_clicked`, отдельно показывает запуски без финального статуса и мультивыбор, добавляет кликабельную ссылку `Полный дашборд с графиками (открывается только с vpn)`, а затем отправляет сообщение через Telegram Bot API. Заголовок отчета отправляется жирным через HTML-разметку Telegram.

## Telegram-команда `/today`

В проекте есть Vercel endpoint `api/telegram.js` для команды `/today`. Он принимает webhook от Telegram, проверяет секретный заголовок Telegram, проверяет `TELEGRAM_CHAT_ID`, игнорирует чужие сообщения и по команде `/today` отправляет отчет за текущий день по Москве.

Для Vercel нужно добавить Environment Variables:

- `POSTHOG_PERSONAL_API_KEY` — PostHog Personal API key для чтения аналитики;
- `POSTHOG_HOST` — `https://eu.posthog.com`;
- `POSTHOG_PROJECT_ID` — `184090`;
- `POSTHOG_DASHBOARD_URL` — `https://eu.posthog.com/project/184090/dashboard/695809`;
- `TELEGRAM_BOT_TOKEN` — токен Telegram-бота;
- `TELEGRAM_CHAT_ID` — id разрешенного чата;
- `TELEGRAM_WEBHOOK_SECRET` — длинная случайная строка для проверки webhook.

После деплоя Vercel webhook нужно привязать к Telegram:

```text
https://api.telegram.org/bot<TELEGRAM_BOT_TOKEN>/setWebhook?url=https://<VERCEL_URL>/api/telegram&secret_token=<TELEGRAM_WEBHOOK_SECRET>
```

После этого команда `/today` в Telegram должна присылать отчет за текущий день. Ежедневный отчет за вчера отправляет Vercel Cron.

Рабочие инструкции по Telegram Bot API, webhook, командам меню, проверкам и сбросу настроек лежат в `docs/telegram-bot-guide.md`.

## Beta-публикация и будущее

Эта версия готовится как private organization beta для внутреннего тестирования. В `manifest.json` название временно указано как `Чистовик — Internal Beta`, чтобы beta-версия не смешивалась с будущей публичной версией.

Перед публичной публикацией в Figma Community нужно будет отдельно подготовить production-метаданные: заменить название, `id`, иконку, обложку и описание. Beta-название и beta-идентификаторы не должны переехать в публичный Community-релиз.

## Как обрабатывается выделение

1. Если выбран текстовый слой, обрабатывается только он.
2. Если выбран фрейм, группа, компонент или другой контейнер, обрабатываются все текстовые слои внутри него рекурсивно.
3. Если ничего не выбрано, обрабатываются все текстовые слои на текущей странице.
4. Заблокированные тексты и тексты внутри заблокированных родителей пропускаются, если в UI не включен чекбокс `Обработать даже слои с замочком`.
5. Скрытые тексты и тексты внутри скрытых родителей пропускаются, если в UI не включен чекбокс `Обработать даже скрытые слои`.

## Уведомления

- Если были изменения и ничего не было пропущено: `Теперь всё чисто 🔥🔥🔥`
- Если менять было нечего и ничего не было пропущено: `Всё уже было чисто 👌`
- Если были изменения, но часть текстов пропущена из-за замков: `Замочки не тронуты, в остальном — теперь всё чисто 🔥🔥🔥`
- Если изменений не было, но часть текстов пропущена из-за замков: `Замочки не тронуты, а остальное уже было чисто 👌`
- Если были изменения, но часть текстов пропущена из-за скрытия: `Скрытые слои не тронуты, в остальном — теперь всё чисто 🔥🔥🔥`
- Если изменений не было, но часть текстов пропущена из-за скрытия: `Скрытые слои не тронуты, а остальное уже было чисто 👌`
- Если были изменения, но часть текстов пропущена из-за замков и скрытия: `Замочки и скрытые слои не тронуты, в остальном — теперь всё чисто 🔥🔥🔥`
- Если изменений не было, но часть текстов пропущена из-за замков и скрытия: `Замочки и скрытые слои не тронуты, а остальное уже было чисто 👌`
- Если произошла ошибка: `Ой, не получилось почистить 🛑`

## Правила типографики

Правила применяются в строгом порядке: сначала чистятся пробелы, затем кавычки и пунктуация, математические примеры, тире и дефисы, телефоны, числа и деньги, сокращения, неразрывные пробелы, математические знаки и символы. В конце дополнительно собираются обратно годы, если рядом с `©` или `г.` они были ошибочно разбиты пробелом.

### 1. Пробелы

- Двойные пробелы заменяются одним.
- Пробелы в начале и конце каждой строки удаляются.
- Пробелы перед `.`, `…`, `:`, `,`, `;`, `?`, `!`, `»`, `)`, `]` удаляются.
- Пробелы после `«`, `(`, `[` удаляются.
- Пробел между числом и процентом удаляется: `50 %` -> `50%`.
- Пробел между `~` и числом или словом удаляется: `~ 100` -> `~100`.

### 2. Кавычки и пунктуация

- Для кириллицы используются елочки первого уровня: `«...»`.
- Для кириллицы внутри елочек используются лапки второго уровня: `„...“`.
- Если вне кавычек есть кириллица, кавычки оформляются по русским правилам даже вокруг латинских слов: `В тексте есть «Design System»`.
- Если вне кавычек латиница, используются прямые двойные кавычки первого уровня и одинарные кавычки второго уровня, даже вокруг русского слова: `The word "привет"`.
- Кавычка сначала определяется по роли в позиции: открывающая или закрывающая. Только после этого выбирается символ по языку локального контекста и уровню вложенности.
- Для кавычек верхнего уровня язык определяется по строке и по тексту вне всей цитаты верхнего уровня. Текст внутри цитаты и внутри вложенных кавычек не должен перевешивать внешний контекст: `Он сказал "Use 'clean typography' mode"` -> `Он сказал «Use „clean typography“ mode»`, но `He said "Use 'clean typography' mode"` остается английской моделью.
- Вложенные кавычки наследуют язык внешних кавычек. Неправильное направление типографских кавычек исправляется по позиции: `»текст»` -> `«текст»`, `«текст„` -> `«текст“`.
- В режиме `Для разработки` звездочки на местах неразрывных пробелов учитываются как пробельный контекст для кавычек, чтобы повторный запуск не менял уже правильно оформленные кавычки.
- Вложенная русская прямая речь приводится к виду: `«„Как это скучно!“ — воскликнул я невольно».`
- Точка, запятая, точка с запятой и двоеточие выносятся за закрывающую кавычку: `Он сказал «привет».`
- Многоточие не переносится через закрывающую кавычку и остается там, где стояло до обработки: `«привет…»` или `«привет»…`.
- Вопросительный и восклицательный знаки переносятся внутрь закрывающей кавычки: `Он спросил «как дела?»`
- Если внутри закрывающей кавычки уже стоит `?` или `!`, точка сразу после кавычек удаляется: `«как дела?».` -> `«как дела?»`.
- Три точки заменяются многоточием: `...` -> `…`.
- Повторные восклицательные знаки схлопываются: `!!` -> `!`.
- Повторные вопросительные знаки схлопываются: `??` -> `?`.
- Сочетание `!?` нормализуется в `?!`.

### 3. Тире и дефисы

- Дефис или среднее тире между словами с пробелами заменяется длинным тире с пробелами: `слово - слово` -> `слово — слово`.
- Перед длинным тире ставится неразрывный пробел на этапе обработки неразрывных пробелов.
- Дефис между буквами заменяется неразрывным дефисом: `кто-то` -> `кто‑то`.
- Некоторые устойчивые сочетания с ошибочным пробелом собираются с неразрывным дефисом: `из за` -> `из‑за`, `из под` -> `из‑под`, `кто то` -> `кто‑то`, `что либо` -> `что‑либо`, `где нибудь` -> `где‑нибудь`, `кое как` -> `кое‑как`, `все таки` -> `все‑таки`, `всё таки` -> `всё‑таки`.
- Простые компактные диапазоны оформляются длинным тире без пробелов: `5-10 кг`, `5 – 10 кг`, `5 — 10 кг` -> `5—10 кг`; `12:25-13:35` -> `12:25—13:35`; `30.04-12.05` -> `30.04—12.05`; `2001-2019` -> `2001—2019`.
- Составные диапазоны оформляются длинным тире с пробелами вокруг; перед тире затем применяется общее правило неразрывного пробела: `2 000-4 000` -> `2 000 — 4 000`; `1 января-15 августа 2018 года` -> `1 января — 15 августа 2018 года`; `2016-н. в.` -> `2016 — н. в.`.
- Диапазоны с римскими цифрами в ожидаемом контексте оформляются длинным тире без пробелов: `X-XI век`, `I - III главы`, `разделы IV-VI` -> `X—XI век`, `I—III главы`, `разделы IV—VI`.
- Римскими цифрами считаются только последовательности в верхнем регистре из `I`, `V`, `X`, `L`, `C`, `D`, `M`; нижний регистр не меняется: `x-xi`.
- Римские диапазоны обрабатываются только рядом со словами `век`, `глава`, `часть`, `раздел`, `пункт`, `квартал`, `кв.`, `том`, `параграф` и их поддержанными формами. Без такого контекста новое правило римских диапазонов не ставит длинное тире: `X-X` не становится `X—X`. При этом общее правило неразрывного дефиса внутри слов продолжает работать для кириллицы и латиницы: `USB-C` -> `USB‑C`, `X-ray` -> `X‑ray`.
- В начале строки маркер `-` или `–` перед пробелом заменяется на длинное тире.

### 4. Телефоны

Телефоном считается последовательность, если она:

- начинается с `+7`, `7` или `8`;
- содержит ровно 11 цифр после удаления пробелов, скобок и дефисов;
- не стоит после знака `№`;
- не стоит перед `₽`, `$` или `€`.

Форматирование:

- все разделители внутри номера удаляются;
- код оператора отделяется неразрывными пробелами;
- группы основного номера отделяются неразрывными дефисами;
- номер с `7` получает плюс, номер с `8` остается с восьмеркой.

Примеры:

- `79001234567` -> `+7 900 123‑45‑67`
- `+7 (900) 123-45-67` -> `+7 900 123‑45‑67`
- `8 900 123 45 67` -> `8 900 123‑45‑67`

Длинные числа, которые не являются телефоном, форматируются как обычные числа: `812345678901234` -> `812 345 678 901 234`.

### 5. Числа, годы и деньги

- Длинные числа разбиваются на разряды неразрывными пробелами: `1000000` -> `1 000 000`.
- Десятичная точка заменяется запятой: `2000.35` -> `2 000,35`.
- Английская запись разрядов и дробной части приводится к русской: `62,226,338.00 ₽` -> `62 226 338,00 ₽`.
- Между числом и валютой, месяцем, единицей измерения или денежно-числовым сокращением ставится неразрывный пробел: `1000 руб.` -> `1 000 руб.`, `1.5 м` -> `1,5 м`, `30 сентября` -> `30 сентября`.
- Версии и IP-адреса не считаются десятичными дробями и не меняются: `v2.0.1`, `192.168.0.1`.
- Числа внутри кодов и слов с дефисом не разбиваются на разряды: `SALE-2026`.
- Даты в формате `10.04` и `10.04.2025` не считаются десятичными дробями и не разбиваются на разряды. Если год в дате уже разбит, он собирается обратно: `10.04.2 025` -> `10.04.2025`.
- Исключение для коротких дат: если после `10.04` стоит единица измерения, денежное сокращение или знак валюты, это считается десятичным числом: `10.04 кг` -> `10,04 кг`, `10.04 ₽` -> `10,04 ₽`.
- Точный список обозначений, которые после короткой даты превращают ее в десятичное число: `₽`, `$`, `€`, `%`, `руб.`, `коп.`, `тыс.`, `млн`, `млрд`, `трлн`, `м`, `км`, `кг`, `г`, `мм`, `см`, `л`, `мл`, `шт.`, `с`, `сек.`, `мин.`, `мес.`, `кв.`, `куб.`.
- Числа после `№` и `§` не разбиваются на разряды: `№ 12345`.
- Годы от `1000` до `2099` не разбиваются рядом с `г.`, `год`, `году`, `в`, `с`, `по`, `©` и `(c)`: `2024 г.`, `© 2024`.
- Между `©` и годом ставится неразрывный пробел: `©2025` -> `© 2025`.
- Уже испорченный год рядом с копирайтом или `г.` собирается обратно: `© 2 024` -> `© 2024`.
- После числа в `руб.` и `коп.` сохраняется или добавляется точка: `1000 руб 20 коп` -> `1 000 руб. 20 коп.`
- Спецслучай аренды или подписки сохраняется без пробела и точки после `мес`: `5000 ₽/мес` -> `5 000 ₽/мес`.

- `н. в.`
- `P. S.`
- `P. P. S.`
- `мес.`, кроме случая `₽/мес`

Для сокращения `под.` точка добавляется только если сокращение уже написано с точкой или стоит перед номером: `под 2` -> `под. 2`. Обычный предлог не меняется: `под столом`.

Составные сокращения площади и объема:

- `кв. м` -> `кв. м`
- `куб. м` -> `куб. м`

У `м` в этих сочетаниях точка не ставится.

Между сокращениями `д.`, `стр.`, `кв.` и следующим числом ставится неразрывный пробел: `д. 5` -> `д. 5`, `стр. 10` -> `стр. 10`, `кв. 7` -> `кв. 7`.

Без точки нормализуются:

- `млн`
- `млрд`
- `трлн`
- `б-р`
- `пр-т`
- единицы после числа: `с`, `м`, `км`, `кг`, `мм`, `см`, `л`, `мл`

Если точка после сокращения без точки, единицы измерения или выражения вроде `₽/мес` является концом предложения, она сохраняется. Новым предложением считается ситуация, когда после точки идет пробел и слово с большой буквы в той же строке. Перенос строки и конец текста сами по себе новым предложением не считаются.

Для сокращений без точки точка в конце строки или текста считается лишней точкой сокращения и удаляется: `5 млрд.` -> `5 млрд`, `5 кг.` -> `5 кг`, `5000 ₽/мес.` -> `5 000 ₽/мес`.

Примеры с точкой конца предложения:

- `100 млн. Компания растёт.` -> `100 млн. Компания растёт.`
- `5 кг. Доставим завтра.` -> `5 кг. Доставим завтра.`
- `5000 ₽/мес. Следующий платёж завтра.` -> `5 000 ₽/мес. Следующий платёж завтра.`

Примеры с лишней точкой сокращения:

- `100 млн. рублей.` -> `100 млн рублей.`
- `5 кг. товара.` -> `5 кг товара.`
- `5000 ₽/мес доступна всем.` -> `5 000 ₽/мес доступна всем.`
- `5 кг.` -> `5 кг`
- `5000 ₽/мес.` -> `5 000 ₽/мес`

Примеры:

- `100 млн.` -> `100 млн`
- `5 млрд.` -> `5 млрд`
- `100 тыс` -> `100 тыс.`
- `20 кв м` -> `20 кв. м`
- `6 мес` -> `6 мес.`

### 7. Неразрывные пробелы

Неразрывный пробел `U+00A0` ставится:

- перед длинным тире: `Москва — столица`;
- после отдельных кириллических слов из 1–2 букв, но не после части слова через дефис: `что‑то на Авито`;
- в инициалах: `И.И. Иванов` -> `И. И. Иванов`, `И.Иванов` -> `И. Иванов`;
- после `№` и `§` перед числом;
- между числом и месяцем, единицей измерения или денежно-числовым сокращением: `30 сентября`, `5 кг`, `100 млн`, `20 °C`;
- между словами из whitelist и следующим числом или компактным диапазоном с процентом: `скидка 5%` -> `скидка 5%`, `скидка 5-10%` -> `скидка 5—10%`, `кэшбэк 10%` -> `кэшбэк 10%`, `НДС 20%` -> `НДС 20%`. Правило не применяется ко всем словам подряд: `оборот 7%` остается с обычным пробелом;
- между обычным числом и следующим словом, если число не является телефоном, номером после `№`, параграфом после `§`, годом или частью кода: `15 завтра` -> `15 завтра`, `10000 клиентов` -> `10 000 клиентов`;
- неразрывный пробел слева от короткого слова не мешает этому правилу: `По 2000 человек` -> `По 2 000 человек`;
- если число уже связано слева числовым правилом, дополнительная связка со следующим словом не добавляется: `д. 5 живёт` -> `д. 5 живёт`, `г. 10000 жителей` -> `г. 10 000 жителей`;
- перед частицами `ли`, `же`, `бы`, `ль`, `ж`, `б`: `далеко ли` -> `далеко ли`, `он же` -> `он же`, `сказал бы` -> `сказал бы`, `можно ль` -> `можно ль`;
- после частиц `ли`, `же`, `бы`, `ль`, `ж`, `б` правило коротких слов не ставит неразрывный пробел: `далеко ли холодно` -> `далеко ли холодно`;
- после дат в формате `9.05`, `09.05`, `9.05.2026` и `09.05.2026` неразрывный пробел не ставится;

### 8. Математика и символы

- Математические примеры с минимум двумя числами и оператором `+`, `*`, `×`, `x`, `х`, `/` или `=` оформляются с неразрывными пробелами вокруг знаков: `10 x 20 = 200` -> `10 × 20 = 200`.
- `x`, `х` или `*` между числами заменяются знаком умножения: `2*2` -> `2 × 2`, `2 * 2` -> `2 × 2`.
- В примерах сложения, умножения, деления и равенства вокруг знаков ставятся неразрывные пробелы: `5 + 5` -> `5 + 5`, `6 / 2 = 3` -> `6 / 2 = 3`.
- Дефис или тире между числами считается вычитанием только в математическом контексте: если в выражении есть `=`, другой математический оператор, больше одного оператора или унарный минус перед первым числом.
- Для вычитания и отрицательных чисел используется знак минус `−`, а не дефис или среднее тире: `10 - 5 = 5` -> `10 − 5 = 5`, `−10 + 5 = −5` -> `−10 + 5 = −5`.
- Отрицательное число пишется слитно со знаком минуса: `-10` -> `−10`.
- Два компактных числа через дефис или тире без математического контекста остаются диапазоном с длинным тире: `5 - 5` -> `5—5`, `10-20` -> `10—20`.
- Телефоны, даты, версии, IP, URL, email, переменные и промокоды не считаются математическими примерами.
- Отдельные дроби заменяются символами: `1/2` -> `½`, `1/4` -> `¼`, `3/4` -> `¾`.
- Дроби внутри слов и версий не трогаются: `v1/2`.
- Температура нормализуется с неразрывным пробелом и знаком градуса: `20C`, `20 C`, `20 °C` -> `20 °C`; `100F` -> `100 °F`.
- `(c)` заменяется на `©`.
- `(tm)` заменяется на `™`.
- `(r)` заменяется на `®`.
- `->` и `=>` заменяются на `→`.
- Диапазоны температур со знаками `+` или `−` разделяются многоточием: `+5-+10 C`, `+5...+10 °C` -> `+5…+10 °C`.

## Как запустить локально

1. Установить зависимости:

```bash
npm install
```

2. Запустить сборку в watch-режиме:

```bash
npm run dev
```

3. В Figma Desktop открыть:

`Plugins` -> `Development` -> `Import plugin from manifest...`

4. Выбрать файл `manifest.json` из корня проекта.

После этого Figma будет запускать собранный файл `dist/code.js`. Если `npm run dev` продолжает работать, изменения из `src/code.ts` будут пересобираться автоматически.

## Структура проекта

- `src/code.ts` — исходный TypeScript-код плагина.
- `src/ui.html` — интерфейс настроек, который Figma получает как готовый HTML.

- `dist/code.js` — собранный JavaScript, который запускает Figma.
- `manifest.json` — манифест Figma-плагина, указывает на `dist/code.js`.

- `assets/plugin-icon.svg` — иконка плагина для публикации и локального предпросмотра. В `manifest.json` она не подключается: Figma задает иконку плагина через окно публикации или обновления ресурса.

## План развития

Текущее состояние проекта — рабочий быстрый запуск и рабочий интерфейс настроек. Команда `Запустить типограф` не открывает UI, обрабатывает текст на текущей странице и показывает результат через `figma.notify`.

Команда `⭐ Продвинутые фичи` открывает UI. Вкладка `Типограф` запускает режим `Для красоты` или режим `Для разработки` и передает в ядро настройки `Обработать даже слои с замочком`, `Обработать даже скрытые слои` и `Считать все одиночные * пробелами`. После обработки из окна настроек UI остается открытым, чтобы пользователь мог запускать следующую обработку без повторного открытия плагина. Дальше интерфейс нужно развивать поэтапно, не ломая быстрый запуск.

### Вкладки UI

В интерфейсе есть три вкладки:

- `Типограф` — рабочая вкладка для запуска обработки. Умеет запускать режимы `Для красоты` и `Для разработки`, а также управлять обработкой заблокированных, скрытых слоев и разбором существующих звездочек в режиме разработки.
- `Правила` — короткое описание правил типографики, оформленное аккордеонами. Аккордеон `Обычные пробелы` стоит после `Неразрывные пробелы`, чтобы было ясно, почему в правилах есть несколько тем про пробелы.
- `О плагине` — текстовый блок о плагине, авторе и ссылка на канал.

Контент вкладок отделен от рабочей логики плагина, чтобы можно было спокойно править тексты и экспериментировать с подачей без риска сломать типографику.

Текущая структура контента:

- `src/ui-content.js` — единый источник текстов вкладок, пунктов правил, подписей режимов и кнопок.
- `src/ui.html` — готовый HTML для Figma. В нем есть размеченные блоки `chistovik-content:*`, которые обновляет скрипт синхронизации.
- `scripts/sync-ui-content.js` — генератор, который переносит данные из `src/ui-content.js` в `src/ui.html`.
- `scripts/send-daily-analytics.js` — скрипт ежедневного Telegram-отчета по PostHog-аналитике.
- `scripts/lib/analytics-report.js` — общая логика PostHog-запросов, форматирования Telegram-сообщений и отправки сообщений.
- `api/telegram.js` — Vercel endpoint для Telegram webhook и команды `/today`.
- `api/daily-analytics.js` — Vercel endpoint для ежедневного Telegram-отчета за вчера.

Важный технический нюанс: Figma-плагин не читает отдельные локальные файлы контента во время работы. Поэтому контент хранится отдельно для удобного редактирования, но встраивается в `src/ui.html` перед сборкой. Команды `npm run build` и `npm run dev` сначала запускают `npm run sync-ui`.

### Производительность

В `manifest.json` включено поле:

```json
"documentAccess": "dynamic-page"
```

Это значит, что Figma не должна заранее загружать все страницы файла перед запуском плагина. Команда `⭐ Продвинутые фичи` открывает UI без обхода документа. Если пользователь выбрал текстовый слой, фрейм, группу, компонент или другой контейнер, плагин обходит только выделение и не вызывает `figma.currentPage.loadAsync()` заранее. Если ничего не выбрано, плагин явно загружает текущую страницу через `figma.currentPage.loadAsync()` и ищет все текстовые слои через `findAllWithCriteria({ types: ["TEXT"] })`.

Для ускорения обхода включается `figma.skipInvisibleInstanceChildren = true`, если пользователь не просит обрабатывать скрытые слои. Это позволяет не заходить в скрытые слои внутри инстансов.

### Режимы типографа

Поддерживаются два режима:

- `Для красоты` — основной режим, текущая логика. В тексте используются настоящие неразрывные пробелы `U+00A0`.
- `Для разработки` — типографика применяется так же, но вместо неразрывных пробелов в текст вставляется символ `*`.

Режимы должны отличаться только способом отображения неразрывного пробела. Все остальные правила типографики остаются одинаковыми.

В режиме `Для разработки`:

- все позиции, где в основном режиме должен быть `U+00A0`, отображаются символом `*`;
- типографские `*` окрашиваются в `#FF4053`;
- пользовательские `*`, не связанные с типографикой, не перекрашиваются и не считаются маркерами неразрывного пробела;
- окрашивание сохраняет остальные стили текста настолько аккуратно, насколько позволяет Figma Plugin API.

Если в режиме `Для разработки` включена галочка `Считать все одиночные * пробелами`, плагин проверяет уже существующие пользовательские `*` как возможные места пробелов. Безопасные одиночные `*` превращаются в обычный пробел перед применением правил, а затем типограф сам решает, останется это обычным пробелом или станет красным служебным маркером неразрывного пробела: `Что*нужно` -> `Что нужно`, `их*комбинации` -> `их*комбинации` с красной служебной звездочкой. Небезопасные `*` в масках, сериях, математике, markdown, латинице, кодах и сносках остаются пользовательскими.

При переключении из режима `Для разработки` в режим `Для красоты` плагин не заменяет все `*` на неразрывные пробелы. Он:

1. Берет текущий текст.
2. Получает список уже существующих development-маркеров по сохраненным plugin data и по стилю/цвету `#FF4053`.
3. Восстанавливает в пробел только те `*`, чьи индексы пришли в `existingDevelopmentMarkerIndexes`.
4. Повторно прогоняет правила типографики.
5. Остальные `*` оставляет пользовательскими символами без угадывания по контексту.

Это нужно, чтобы не ломать маскированные данные, математические выражения, сноски и любые пользовательские звёздочки.

Важно: служебными считаются только те `*`, которые уже были созданы плагином как development-маркеры и найдены по сохраненным индексам или по стилю/цвету. Plugin data используется только если сохраненный текст полностью совпадает с текущим текстом ноды; если текст вручную изменили, старые индексы игнорируются. Если индекс маркера не передан, `*` остается пользовательским символом и может участвовать в других правилах, например `2 * 2 = 4` должно стать `2 × 2 = 4`.

Повторный запуск режима `Для разработки` дополнительно защищает устойчивые уже-типографированные паттерны: телефоны, суммы, годы рядом с `©` и разрядные числа со `*` не должны попадать в математическое правило и превращаться в умножение. Это ограниченная защита форматов вроде `+7*900*123‑45‑67`, `1*000*₽`, `©*2025`, `812*345*678*901*234`; произвольные пользовательские `*` не удаляются и не превращаются в пробел.

Телефоноподобные последовательности после `№` считаются номером заказа, а не телефоном: `№ +7 900 123-45-67` не приводится к телефонному формату и при повторном запуске `Для разработки` не превращается в математическое выражение.

### Настройки запуска

В коде есть единый объект настроек `PluginRunOptions`:

```ts
{
  mode: "beauty" | "development",
  processHiddenNodes: boolean,
  processLockedNodes: boolean,
  recolorExistingAsterisks: boolean
}
```

Текущие значения по умолчанию:

- `mode: "beauty"`;
- `processHiddenNodes: false`;
- `processLockedNodes: false`.
- `recolorExistingAsterisks: false`.

Чекбокс `Обработать даже слои с замочком` выключен по умолчанию. Замок в макете считается намерением дизайнера не менять слой. Если чекбокс включен, обработка затрагивает заблокированные тексты и тексты внутри заблокированных родителей.

Чекбокс `Обработать даже скрытые слои` выключен по умолчанию. Если чекбокс включен, обработка затрагивает скрытые тексты и тексты внутри скрытых родителей, а также отключает пропуск скрытых слоев внутри инстансов.

Чекбокс `Считать все одиночные * пробелами` доступен только в режиме `Для разработки` и выключен по умолчанию. Если он включен, безопасные одиночные `*` разбираются как пробельные кандидаты: часть становится обычными пробелами, а часть — служебными development-маркерами, если на этом месте по правилам нужен неразрывный пробел. После такого запуска режим `Для красоты` воспринимает только красные служебные `*` как неразрывные пробелы.

### План UI-отчета об ошибках

Ближайшая крупная доработка диагностики — пользовательский UI-отчет об ошибках обработки. Если хотя бы один текстовый слой не удалось обработать, плагин должен продолжить обход остальных слоев, обработать все возможное, а затем открыть UI с отчетом вместо дефолтного уведомления `Ой, не получилось почистить 🛑`.

В отчете слева должен быть список текстовых слоев с ошибками и счетчик ошибок. Основной идентификатор в списке — `textPreview`, короткий фрагмент текста из слоя; название слоя можно не показывать или показывать вторично мелким серым текстом. Справа — подробности выбранной ошибки: что случилось, технический код, понятное объяснение и что пользователь может проверить руками. При клике на слой плагин должен выделять его в Figma и приближать холст через `figma.viewport.scrollAndZoomIntoView([node])`, чтобы пользователь мог быстро найти проблемное место. Для `single_text` используется тот же отчет, но список слева содержит один элемент.

Эта доработка должна быть локальной пользовательской диагностикой внутри Figma. Отправку новых типов ошибок в PostHog и Telegram пока откладываем; обсуждение и согласованные тексты сохранены в `docs/error-monitoring.md`.

До интеграции в Figma нужно сделать HTML-предпросмотр `preview/error-report-preview.html` с фейковыми ошибками, чтобы согласовать тексты, состояния и дизайн окна без необходимости ловить реальные ошибки в макете. Для ручного тестирования в Figma позже нужен локальный test mode: он выключен по умолчанию, работает только при явном dev-флаге и только с явными маркерами вроде `[chistovik-dev-error:load_fonts]`. В опубликованной версии такие маркеры должны оставаться обычным текстом.

Критическое требование: при реализации UI-отчета нельзя менять правила типографики, порядок правил, регулярные выражения, ожидаемые результаты тестов, поведение режимов `Для красоты` и `Для разработки`, стратегию development-маркеров и сохранение mixed styles. Новый UI должен быть только диагностическим слоем вокруг текущей обработки.

Подробный план хранится в `docs/error-ui-plan.md`. Визуальные правила всего интерфейса, включая окно ошибок, зафиксированы в `docs/ui-design-system.md`; при дальнейшей реализации нужно сверяться с ними, чтобы не уехать от дизайн-системы текущего плагина.

### Порядок следующих этапов

1. Реализовать UI-отчет об ошибках обработки по плану из `docs/error-ui-plan.md`, не меняя правила типографики.
2. Сохранять стабильность быстрого запуска: команда `Запустить типограф` должна работать без UI, если ошибок нет; UI открывается только для отчета об ошибках.
3. После текстовых правок в `src/ui-content.js` запускать `npm run build`, чтобы обновить `src/ui.html` и проверить TypeScript.
4. После каждого этапа проверять быстрый запуск в Figma и сохранять версию отдельным русским коммитом.

### Требования к будущей архитектуре

- Рабочая типографика должна оставаться отделенной от UI.
- Правила типографики считаются критической зоной: нельзя менять `cleanTypography`, порядок правил, регулярные выражения, исключения и тестовые ожидания ради UI-отчета, аналитики или Telegram-логов.
- Аналитика должна оставаться отделенной от правил типографики и не должна менять поведение `cleanTypography`.
- Ежедневный Telegram-отчет должен жить отдельно от Figma-плагина: через Vercel Cron и `api/daily-analytics.js`, без влияния на запуск плагина в Figma.
- Telegram-команды должны жить отдельно от Figma-плагина: через Vercel endpoint `api/telegram.js`, без изменений `src/code.ts`, `dist/code.js`, `src/ui.html`, `src/ui-content.js` и `manifest.json`.
- Быстрый запуск не должен зависеть от состояния открытого интерфейса.
- UI должен передавать настройки в основную логику сообщением, а не дублировать правила обработки.
- Тексты вкладок должны храниться в `src/ui-content.js`, отдельно от ядра типографики.
- Перед изменением текстов вкладки `Правила` нужно сначала прислать пользователю новые формулировки и вносить их только после явного согласования.
- Не загружать все страницы файла без необходимости. Плагин должен оставаться совместимым с `documentAccess: "dynamic-page"` и работать только с текущей страницей.
- Не использовать `typograf`, если его правила могут конфликтовать с кастомной логикой.
- Не пересоздавать текстовые слои и не заменять их новыми нодами.
- Продолжать менять текст только через `textNode.characters = newText`.
- Не отправлять в аналитику текст макета, названия файлов, страниц, слоев, email, имя пользователя или Figma user id.
- Ошибки аналитики не должны блокировать работу плагина и не должны показываться пользователю.
- После любых замен длины текста проверять mixed styles и цветовые диапазоны.
- Перед коммитом нужно написать пользователю полный список изменений, которые попадут в коммит: не только запрошенное изменение, но и все сопутствующие изменения поведения, текстов, уведомлений, настроек, README, сборки и manifest. Если меняются уведомления, нужно явно показать новые тексты уведомлений до коммита.
- Каждую версию сохранять отдельным локальным git-коммитом на русском языке с понятным описанием изменений.
- Не делать `git push` и не пытаться пушить через GitHub API. Даже если пользователь просит «сделай пуш», это означает: подготовить локальный коммит, а push origin пользователь нажмет руками в GitHub Desktop.
- После каждого коммита дополнять `docs/commit-log.md`: указывать код коммита, что изменилось, зачем это было сделано, какие проверки прошли и что важно не потерять в следующем чате.
- После каждого этапа давать инструкцию, как проверить в Figma, что ничего не сломалось.

## Журнал коммитов

Полный журнал коммитов вынесен в `docs/commit-log.md`, чтобы README не разрастался. После каждого нового локального коммита нужно добавлять запись на русском языке именно туда: с кодом коммита, смыслом изменений, затронутыми файлами, проверками и важными договоренностями для следующего чата.

## Заметки для продолжения разработки

- Главная функция обработки текста: `cleanTypography`.
- Основной запуск проходит через объект настроек `PluginRunOptions`: режим `beauty` или `development`, флаг `processLockedNodes`, флаг `processHiddenNodes` и флаг `recolorExistingAsterisks`.
- Порядок правил важен и зафиксирован внутри `cleanTypography`.
- При возврате из режима `Для разработки` в `Для красоты` нельзя угадывать служебные `*` по общим типографским правилам: источником истины для обычных маркеров являются `existingDevelopmentMarkerIndexes`, собранные по plugin data и по стилю/цвету. Сохраненные plugin data можно использовать только при полном совпадении сохраненного текста с текущим текстом ноды. Отдельно разрешена только ограниченная защита уже-типографированных телефонов, сумм и разрядных чисел, чтобы повторный запуск development не превращал их в математику.
- Не менять стратегию на пересоздание текстовых слоев: это сломает стили и связи в макете.
- Не обрабатывать заблокированные слои по умолчанию: это считается намерением дизайнера не менять слой. В UI уже есть чекбокс `Обработать даже слои с замочком`, который передает в ядро `processLockedNodes`.
- Не обрабатывать скрытые слои по умолчанию. В UI уже есть чекбокс `Обработать даже скрытые слои`, который передает в ядро `processHiddenNodes`.
- Если добавляются новые правила, нужно проверять повторный запуск плагина: результат должен быть идемпотентным.
- Для правил с заменой длины текста нужно следить за сохранением mixed styles. Сейчас стили сопоставляются через карту старых и новых символов, а затем восстанавливаются диапазонами.
- Коммиты нужно писать на русском языке. Сообщение коммита должно объяснять суть версии так, чтобы через время было понятно, какие изменения вошли и к какой версии можно откатиться.
- Push origin делает пользователь вручную через GitHub Desktop. Не тратить время и токены на попытки пуша из чата.
- После правок кода запускать:

```bash
npm run build
```
