# План внедрения аналитики

Документ фиксирует согласованный подход к аналитике для Figma-плагина «Чистовик». Он нужен как рабочая спецификация для следующей сессии разработки: что считать, куда отправлять, как не повлиять на пользователя и какие границы нельзя нарушать.

## Главный принцип

Аналитика должна быть встроена «под капотом» и не должна менять поведение плагина для пользователя.

Жесткое требование: внедрение аналитики не должно затронуть текущую логику типографики, правила обработки текста, порядок правил, сохранение mixed styles, работу development-маркеров, уведомления и сценарии запуска.

Нельзя:

- менять правила внутри `cleanTypography`;
- менять порядок существующих типографских замен;
- менять тексты и логику текущих `figma.notify`;
- менять обработку заблокированных и скрытых слоев;
- менять стратегию сохранения mixed styles;
- пересоздавать текстовые слои;
- отправлять в аналитику текст макета, названия слоев, названия файлов или персональные данные;
- делать работу типографа зависимой от сети или ответа аналитического сервиса.

Можно:

- добавить отдельный слой аналитики вокруг сценариев запуска;
- считать параметры запуска, итоговые счетчики и технические ошибки;
- отправлять события асинхронно и без влияния на результат обработки;
- документировать наличие анонимной технической аналитики в README и будущем описании плагина.

## Пользовательский опыт

Пользователь не должен заметить появление аналитики в интерфейсе или скорости работы.

Требования:

- не добавлять чекбокс отказа от аналитики;
- не добавлять новые модалки, предупреждения или настройки;
- не показывать уведомления об успешной или неуспешной отправке аналитики;
- не задерживать результат типографа из-за аналитики;
- если аналитический сервис недоступен, плагин работает как раньше;
- если отправка события упала с ошибкой, ошибка тихо гасится и не влияет на `figma.notify`;
- аналитика должна быть описана только в документации и будущем публичном описании плагина.

## Сервис аналитики

Рекомендуемый первый сервис: PostHog Cloud.

Причины:

- есть бесплатный тариф, достаточный для небольшого плагина;
- на момент проверки у PostHog есть бесплатный лимит 1 млн analytics events в месяц;
- есть отдельный бесплатный лимит для error tracking;
- есть графики, дашборды, breakdown по свойствам событий, фильтры и retention;
- можно отправлять события напрямую через HTTP Capture API без тяжелой SDK;
- в будущем можно заменить сервис, если оставить внутри плагина тонкий слой `trackEvent(...)`.

Альтернативы:

- Umami: проще и больше про веб-аналитику; хуже подходит для сложных событий с режимами, флагами и ошибками.
- Supabase или собственная база: гибко, но придется самостоятельно строить отчеты и графики.
- Собственный endpoint перед PostHog: хороший будущий вариант, но для первой версии добавляет лишнюю инфраструктуру.

## Сетевой доступ Figma

Сейчас в `manifest.json` указано:

```json
"networkAccess": {
  "allowedDomains": ["none"]
}
```

Для аналитики нужно будет заменить `["none"]` на конкретный домен выбранного сервиса, например PostHog:

```json
"networkAccess": {
  "allowedDomains": ["https://eu.i.posthog.com"],
  "reasoning": "Плагин отправляет анонимные технические события использования и ошибки для улучшения продукта."
}
```

Регион PostHog нужно выбрать заранее: EU или US. После выбора домена разрешать только его, не использовать `"*"`.

Важно: список разрешенных доменов отображается на странице плагина в Figma Community, поэтому в описании плагина должна быть прозрачная формулировка, зачем нужен сетевой доступ.

## Идентификация пользователей

Для первой версии использовать анонимный идентификатор:

1. При первом запуске сгенерировать случайный id.
2. Сохранить его в `figma.clientStorage`.
3. Использовать этот id как `distinct_id` для событий аналитики.

Плюсы:

- не нужны дополнительные permissions в `manifest.json`;
- не отправляются имя, email или Figma user id;
- достаточно для оценки уникальных пользователей и повторных запусков.

Минусы:

- если пользователь очистит кэш, id может потеряться;
- если пользователь работает на другом устройстве, он может считаться другим пользователем;
- если изменится `plugin id`, старый `clientStorage` станет недоступен.

Эти минусы приемлемы для первой версии.

## Будущая платная версия

Дальняя цель: возможно, у плагина появится платный тариф и добровольная привязка email.

Текущая анонимная схема этому не мешает, если сразу заложить нейтральный слой идентификации.

В коде лучше использовать не `getAnonymousUserId`, а более общий слой:

```ts
type AnalyticsIdentity = {
  distinctId: string;
  anonymousId: string;
  userId: string | null;
  identityType: "anonymous" | "identified";
};
```

Сейчас:

```ts
{
  distinctId: anonymousId,
  anonymousId,
  userId: null,
  identityType: "anonymous"
}
```

В будущем:

```ts
{
  distinctId: userId,
  anonymousId,
  userId,
  identityType: "identified"
}
```

Email в будущем не стоит отправлять как `distinct_id` в открытом виде. Лучше:

- хранить email на своем backend;
- в аналитику отправлять внутренний account/user id;
- или использовать хэш email, если нужен компромисс.

При появлении аккаунтов можно будет связать старую анонимную историю с новым пользователем через механизм alias/identify выбранного сервиса.

## Что считать

Обязательные метрики:

- количество уникальных пользователей;
- количество ежедневных запусков плагина;
- количество запусков быстрым запуском в режиме по умолчанию;
- количество запусков из настроек в режиме `Для красоты`;
- количество запусков из настроек в режиме `Для разработки`;
- в каждом режиме: сколько запусков было с обработкой заблокированных слоев;
- в каждом режиме: сколько запусков было с обработкой скрытых слоев;
- количество кликов на ссылку на канал;
- количество ошибок плагина;
- ошибки по режимам, настройкам и типу запуска;
- тип области запуска: текстовый слой, контейнер или вся страница.

Дополнительные полезные метрики:

- длительность обработки в миллисекундах;
- количество найденных текстовых слоев;
- количество обработанных текстовых слоев;
- количество измененных текстовых слоев;
- количество пропущенных заблокированных текстов;
- количество пропущенных скрытых текстов;
- запуск с одним выделенным объектом или мультивыбором;
- версия плагина;
- версия схемы аналитики.

## Тип области запуска

Нужно считать, как пользователь запускает типограф:

- `single_text` — выбран один текстовый слой;
- `container` — выбран фрейм, группа, компонент, инстанс, секция или другой контейнер;
- `page` — ничего не выбрано, обрабатывается вся текущая страница;
- `multi_selection` — выбрано несколько объектов.

Эта классификация не должна менять текущую логику выбора текстовых слоев. Она только описывает исходное состояние selection для аналитики.

Если выбран контейнер, не нужно отправлять его имя или структуру. Только тип сценария.

## События

Минимальный набор событий:

- `plugin_run_started`;
- `plugin_run_completed`;
- `plugin_run_failed`;
- `settings_opened`;
- `channel_link_clicked`.

Опционально позже:

- `analytics_identity_created`;
- `analytics_transport_failed`, только если есть смысл диагностировать саму аналитику, но без пользовательских уведомлений.

## Свойства событий запуска

Общие свойства для запусков:

```ts
{
  analytics_schema_version: 1,
  plugin_version: "1.0.0",
  source: "quick_run" | "settings",
  mode: "default" | "beauty" | "development",
  process_locked_nodes: boolean,
  process_hidden_nodes: boolean,
  recolor_existing_asterisks: boolean,
  selection_scope: "single_text" | "container" | "page" | "multi_selection",
  selected_nodes_count: number,
  selected_text_nodes_count: number,
  identity_type: "anonymous" | "identified"
}
```

Дополнительные свойства для `plugin_run_completed`:

```ts
{
  processed_text_nodes_count: number,
  changed_text_nodes_count: number,
  skipped_locked_count: number,
  skipped_hidden_count: number,
  duration_ms: number,
  changed_anything: boolean
  failed_text_nodes_count: number,
  found_text_nodes_count: number
}
```

Дополнительные свойства для `plugin_run_failed`:

```ts
{
  duration_ms: number,
  stage: "load_page" | "collect_nodes" | "load_fonts" | "clean_text" | "apply_text" | "unknown",
  error_name: string,
  error_fingerprint: string,
  failed_text_nodes_count: number | null,
  found_text_nodes_count: number | null,
  processed_text_nodes_count: number | null
}
```

Не отправлять полный stack trace, если он может содержать лишний контекст. Для дашборда достаточно имени ошибки, этапа и короткого fingerprint.

## Событие клика на канал

Для клика по ссылке на канал отправлять:

```ts
{
  analytics_schema_version: 1,
  plugin_version: "1.0.0",
  source: "about_tab",
  link: "channel"
}
```

Не отправлять полный URL, если он не нужен. Лучше использовать стабильный идентификатор `channel`.

Клик должен работать как раньше. Аналитика не должна задерживать открытие ссылки.

## Архитектура внедрения

Добавить отдельные функции аналитики, не смешивая их с правилами типографики:

- `getAnalyticsIdentity()`;
- `trackAnalyticsEvent(name, properties)`;
- `trackRunStarted(context)`;
- `trackRunCompleted(context, result)`;
- `trackRunFailed(context, error)`;
- `classifySelectionScope(selection)`;
- `createErrorFingerprint(error)`.

Все вызовы отправки делать так, чтобы они не блокировали пользовательский сценарий:

```ts
void trackAnalyticsEvent("plugin_run_completed", properties);
```

Внутри `trackAnalyticsEvent` обязательно должен быть собственный `try/catch`.

Пример принципа:

```ts
async function trackAnalyticsEvent(name: string, properties: AnalyticsProperties): Promise<void> {
  try {
    // get identity
    // send fetch
  } catch {
    // Intentionally ignored: analytics must never affect the plugin.
  }
}
```

Не добавлять `await trackAnalyticsEvent(...)` в критический путь перед изменением текста или перед пользовательским уведомлением.

## Формат PostHog Capture payload

События нужно отправлять в PostHog Capture API без кастомного поля `uuid`.

Внутренние id событий вида `evt_<timestamp>_<random>` можно использовать только внутри локальной очереди плагина, например как `QueuedAnalyticsEvent.id`. Их нельзя передавать в PostHog как `uuid`: Capture API принимает отсутствие `uuid` или настоящий UUID-формат, а произвольная строка приводит к `400 Bad Request`.

Безопасный минимум payload:

```ts
{
  api_key: ANALYTICS_PROJECT_TOKEN,
  distinct_id: identity.distinctId,
  event,
  properties: {
    ...properties,
    $geoip_disable: true,
    $process_person_profile: false,
    analytics_schema_version: 1,
    identity_type: identity.identityType,
    plugin_version: ANALYTICS_PLUGIN_VERSION
  },
  timestamp: capturedAt
}
```

Если для дедупликации в PostHog когда-нибудь понадобится `uuid`, его нужно генерировать настоящим UUID v4 и отдельно проверить Capture API на тестовом событии с `is_test_event: true`.

Локальная очередь событий в `figma.clientStorage` должна быть устойчивой к старому формату: при чтении очереди legacy-payload нужно нормализовать перед отправкой, чтобы старое невалидное поле не блокировало новые события.

## Где ставить аналитику

Ставить аналитику вокруг сценариев запуска:

- при быстром запуске команды `Запустить типограф`;
- при открытии окна настроек;
- при запуске режима `Для красоты` из настроек;
- при запуске режима `Для разработки` из настроек;
- при клике на ссылку канала;
- в общем `catch` вокруг выполнения типографа.

Не ставить аналитику внутрь отдельных правил типографики.

Если для счетчиков нужны данные из результата обработки, брать только агрегаты, которые уже есть или могут быть добавлены вокруг обхода нод. Не отправлять содержимое текста.

## Ошибки

Нужно отслеживать ошибки, чтобы понимать, насколько часто плагин падает.

Требования:

- ошибка аналитики не должна считаться ошибкой плагина;
- ошибка типографа должна отправляться как `plugin_run_failed`;
- пользовательское уведомление об ошибке должно остаться текущим: `Ой, не получилось почистить 🛑`;
- если отправить `plugin_run_failed` не удалось, пользователь ничего дополнительного не видит.

Для этапов ошибки использовать грубую классификацию:

- `load_page`;
- `collect_nodes`;
- `load_fonts`;
- `clean_text`;
- `apply_text`;
- `unknown`.

Не отправлять текст, имя слоя, имя файла или страницу.

## Приватность и документация

В README и будущем описании плагина нужно явно описать:

- плагин отправляет анонимные технические события использования;
- аналитика нужна для понимания запусков, режимов, ошибок и кликов по авторской ссылке;
- плагин не отправляет текст макета;
- плагин не отправляет названия файлов, страниц и слоев;
- плагин не отправляет email, имя пользователя или Figma user id;
- события должны отправляться с `$geoip_disable: true`, чтобы PostHog не добавлял город, страну, координаты и другие свойства по IP;
- отказа от аналитики в UI нет.

Если в будущем появится платный тариф и email, это должно быть отдельным изменением с отдельной формулировкой согласия.

## Проверки перед внедрением

Перед изменениями:

- запустить `npm test` на текущем коде и убедиться, что базовые правила проходят;
- зафиксировать, что этот этап не должен менять вкладку `Правила`.

После внедрения:

- `npm test`;
- `npm run build`;
- `git diff --check`;
- ручная проверка быстрого запуска;
- ручная проверка запуска из настроек в режимах `Для красоты` и `Для разработки`;
- ручная проверка, что уведомления не изменились;
- ручная проверка, что при недоступной сети типограф работает;
- ручная проверка клика на канал;
- проверка, что события появились в PostHog;
- проверка, что в событиях нет текста макета, имен файлов, страниц и слоев.

## Проверки на неизменность типографики

Особое внимание:

- существующие тесты типографики должны проходить без изменения ожидаемых результатов;
- не менять тесты ради аналитики, кроме добавления отдельных тестов для аналитических helper-функций, если они появятся;
- если `cleanTypography` или соседняя логика обхода нод будут затронуты технически, перед коммитом отдельно описать почему и что именно изменилось;
- перед коммитом явно написать пользователю: правила типографики не менялись.

## План этапов

1. Утвердить сервис аналитики и регион PostHog: EU или US.
2. Создать проект в PostHog и получить project token.
3. Утвердить финальную event schema.
4. Добавить слой аналитики и anonymous identity через `figma.clientStorage`.
5. Добавить fire-and-forget отправку событий через PostHog Capture API.
6. Добавить классификацию `selection_scope`.
7. Добавить события запусков, ошибок, открытия настроек и клика на канал.
8. Обновить `manifest.json`: разрешить только нужный домен аналитики.
9. Обновить README: приватность, сетевой доступ, схема аналитики, требования к будущим изменениям.
10. Запустить проверки.
11. Показать пользователю полный список изменений перед коммитом.
12. Сделать отдельный русский локальный коммит.
13. Добавить запись в `Журнал коммитов`.
14. Дать инструкцию, как проверить в Figma и в PostHog.

## Вопросы перед стартом разработки

Перед реализацией нужно согласовать:

- PostHog EU или PostHog US;
- точный project token и способ его хранения в коде;
- финальные названия событий;
- нужна ли отправка `plugin_run_started` или достаточно `completed` и `failed`;
- нужно ли считать `settings_opened`;
- нужно ли считать количество выбранных объектов при мультивыборе;
- нужно ли в первой версии добавлять отдельные тесты на helper-функции аналитики.
