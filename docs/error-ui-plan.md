# UI-отчет об ошибках обработки

Документ фиксирует новый ближайший план: показывать пользователю конкретные текстовые слои, которые не удалось обработать, прямо в UI плагина. Идея логов и Telegram-типологии ошибок откладывается: сначала нужно сделать полезную диагностику для человека внутри Figma.

## Главная цель

Если при запуске типографа хотя бы один текстовый слой не обработался, пользователь должен увидеть понятный отчет:

- какие текстовые слои упали;
- сколько текстовых слоев не удалось обработать;
- что примерно случилось с каждым слоем;
- как перейти к проблемному слою в макете.

UI должен работать похоже на поиск в Figma: слева список результатов, при клике плагин выделяет слой и приближает к нему холст. Визуальные правила окна зафиксированы в `docs/error-ui-design-system.md`.

## Критическое требование

При реализации этого этапа нельзя менять правила типографики.

Нельзя:

- менять `cleanTypography`;
- менять порядок правил внутри `cleanTypography`;
- менять регулярные выражения, условия и исключения типографики;
- менять ожидаемые результаты тестов типографики ради нового UI;
- менять поведение режимов `Для красоты` и `Для разработки`;
- менять стратегию development-маркеров;
- менять сохранение mixed styles, кроме точечных исправлений, явно связанных с ошибками обработки.

Новый UI-отчет должен быть только слоем диагностики вокруг существующей обработки. Тексты, которые успешно обрабатывались до этого этапа, должны продолжить обрабатываться точно так же.

## Желаемое поведение

Если ошибок нет:

- быстрый запуск работает как сейчас;
- запуск из настроек работает как сейчас;
- показываются текущие успешные уведомления;
- UI не открывается специально ради отчета.

Если ошибки есть:

- плагин продолжает обходить остальные текстовые слои;
- все слои, которые можно обработать, должны быть обработаны;
- после обхода вместо дефолтного уведомления `Ой, не получилось почистить 🛑` открывается UI-отчет;
- отчет показывает список проблемных текстовых слоев и подробности выбранной ошибки;
- пользователь может кликнуть на слой в списке, чтобы Figma выделила его и приблизила холст.

Важно: если часть слоев уже успешно обработалась, это нужно честно показать в отчете, чтобы пользователь понимал, что ошибка не всегда означает полный откат обработки.

## Структура UI

Все кегли, цвета, отступы, состояния строк и кнопки должны соответствовать `docs/error-ui-design-system.md`.

Левая колонка:

- список текстовых слоев с ошибками;
- основной идентификатор ошибки — `textPreview`, короткий фрагмент текста из слоя;
- название слоя можно не показывать или показывать вторично: мелким серым текстом;
- короткое название ошибки;
- выбранный пункт подсвечивается.

Заголовок отчета берет на себя счетчик:

- для ошибок конкретных текстовых слоев: `Не удалось обработать N слой/слоя/слоёв`;
- слово `слой` должно правильно склоняться в зависимости от числа;
- для общей ошибки запуска: `Не удалось запустить типограф`.

Правая колонка:

- название выбранного слоя;
- человекочитаемое название ошибки;
- технический код ошибки;
- понятное объяснение;
- подсказка, что пользователь может проверить руками;
- кнопка или действие `Показать в макете`, если слой можно найти.

## Навигация к слою

При клике на ошибку UI отправляет сообщение в основной код плагина с `nodeId`.

Основной код должен:

1. найти ноду через `figma.getNodeById`;
2. убедиться, что это текстовый слой;
3. выделить его через `figma.currentPage.selection`;
4. вызвать `figma.viewport.scrollAndZoomIntoView([node])`.

Если слой уже удален или недоступен, UI должен показать мягкое сообщение, что перейти к нему больше нельзя.

## Модель ошибки

Для UI нужна внутренняя модель ошибки. Она не обязана сразу уходить в аналитику.

Минимальные поля:

```ts
{
  nodeId: string;
  nodeName: string;
  textPreview: string;
  stage: string;
  title: string;
  description: string;
  suggestion: string;
  technicalCode: string;
  errorName: string;
}
```

Если ошибка произошла внутри правил типографики, позже можно добавить:

```ts
{
  typographyRuleCode: string;
  typographyRuleTitle: string;
}
```

## Первый набор ошибок для UI

На первом этапе достаточно покрыть крупные этапы обработки:

- чтение текста;
- поиск служебных звездочек режима `Для разработки`;
- применение правил типографики;
- загрузка шрифтов;
- сохранение оформления;
- построение карты стилей;
- запись текста;
- восстановление оформления;
- проверка и окраска служебных звездочек;
- сохранение plugin data для служебных звездочек;
- неизвестная ошибка.

Точные редакторские формулировки словаря согласуются отдельно перед внедрением в UI.

## План реализации

1. Убрать зависимость пользовательской диагностики от PostHog и Telegram: сначала делаем локальный UI-отчет.
2. Сделать статический предпросмотр `preview/error-report-preview.html` с фейковыми ошибками, чтобы согласовать тексты, список, правую панель и дизайн до тестирования в Figma.
3. Добавить сбор ошибок по текстовым слоям в `processTextNodes`, не меняя сами правила типографики.
4. Сохранять ошибки в массив, а не только счетчиком `failed`.
5. Продолжать обработку следующих текстовых слоев после ошибки одного слоя.
6. Если ошибок нет, оставить текущие уведомления и закрытие плагина как есть.
7. Если ошибки есть, открывать или переключать UI в режим отчета.
8. Добавить в `src/ui.html` и `src/ui-content.js` состояние отчета об ошибках.
9. Добавить сообщения UI -> code для перехода к проблемному слою.
10. Добавить локальный test mode для искусственных ошибок с двумя обязательными условиями: включенный флаг разработки и явный dev-маркер ошибки в тексте.
11. Добавить проверку перед коммитом, что test mode выключен.
12. Добавить ручную проверку в Figma на текстах с разными искусственными ошибками.
13. Только после стабильного UI отдельно вернуться к вопросу, какие агрегированные данные безопасно отправлять в аналитику.

## Предпросмотр UI

До интеграции в Figma нужно сделать отдельный HTML-предпросмотр:

```text
preview/error-report-preview.html
```

Предпросмотр должен показывать фейковые данные и не зависеть от реальных ошибок Figma. В нем нужно проверить:

- одну ошибку;
- много ошибок;
- длинные названия слоев;
- длинные объяснения;
- выбранное состояние;
- удаленный или недоступный слой;
- ошибку правила типографики;
- пустое состояние, если оно понадобится.

После согласования текстов и дизайна в предпросмотре UI переносится в `src/ui.html` и `src/ui-content.js`.

## Локальный test mode

Для ручного тестирования в Figma нужен механизм искусственных ошибок. Он должен быть безопасным и недоступным в обычной опубликованной версии.

Требования:

- test mode выключен по умолчанию;
- включается только явным dev-флагом в коде, например `ENABLE_ERROR_TEST_MODE = true`;
- перед коммитом флаг должен быть `false`;
- искусственная ошибка срабатывает только если test mode включен и текст содержит явный dev-маркер;
- обычный пользовательский текст не должен случайно запускать ошибку.

Пример формата dev-маркера:

```text
[chistovik-dev-error:load_fonts]
```

Черновой список dev-маркеров:

- `[chistovik-dev-error:read_text]`;
- `[chistovik-dev-error:read_development_markers]`;
- `[chistovik-dev-error:clean_text]`;
- `[chistovik-dev-error:load_fonts]`;
- `[chistovik-dev-error:capture_styles]`;
- `[chistovik-dev-error:build_style_map]`;
- `[chistovik-dev-error:write_text]`;
- `[chistovik-dev-error:restore_styles]`;
- `[chistovik-dev-error:apply_development_marker_styles]`;
- `[chistovik-dev-error:sync_development_marker_data]`.

Эти маркеры нужны только для локального тестирования UI-отчета. В продакшене при выключенном test mode они должны оставаться обычным текстом и не влиять на обработку.

## Проверки

Обязательные проверки после реализации:

- `npm test`;
- `npm run build`;
- `git diff --check`;
- быстрый запуск без ошибок должен работать как раньше;
- запуск из настроек без ошибок должен работать как раньше;
- для `single_text` используется тот же UI-отчет, но список слева содержит один элемент;
- при ошибке одного слоя остальные слои должны обработаться;
- UI-отчет должен открываться только при ошибках;
- клик по ошибке должен выделять слой и приближать холст;
- искусственные dev-ошибки не срабатывают при выключенном test mode;
- перед коммитом test mode выключен;
- список ошибок не должен отправлять в PostHog текст макета, названия слоев или другую приватную информацию.

## Что отложено

Типология ошибок для PostHog, Telegram-отчетов и отдельная справочная команда бота откладываются. Наработки и согласованные тексты сохранены в `docs/error-monitoring.md`, чтобы к ним можно было вернуться после появления UI-отчета.
